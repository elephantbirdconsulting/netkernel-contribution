URL's used

for individuals

The used URL's are composed according to following pattern:

   http://data.kbodata.be/{concept}/{identifier}#id
  • concept: the type of entity e.g organisation, address, ...
  • identifier: the kbo identifier where '.' have been replaced by '_'.

 Examples:

   http://data.kbodata.be/organisation/0308_357_159#id
   http://data.kbodata.be/address/2_189_876_354#id

The full pattern of all entities in the database: ^http://data.kbodata.be/(activity|activitygroup|address|classification|conceptscheme|contacttype|entitycontact|juridicalform|juridicalsituation|language|nace2003|nace2008|organisation|status|typeofaddress|typeofdenomination|typeofenterprise|activity|softwareagent)/(.*)#id$

for classes and properties

We use, except for one, classes and properties from well-know vocabularies. These vocabularies are:

  • schema
  • org
  • rov
  • oslo
  • rdfs
  • rdf
  • vcard
  • skos
  • owl
  • void
  • dcterms
  • foaf
  • kbo
  • locn
  • xsd

We added one property being orgMainActivity with identifier http://data.kbodata.be/def#orgMainActivity. As you might expect the definition can be found at this address.

conneg (content negotiation)

The linked data server supports content negotiation for following formats:

  • html
  • rdf/xml (default)
  • turtle
  • ntriples
  • jsonld

Use when doing a HTTP GET as header:

Accept for a response format as
rdf
*/* rdf
text/html html
application/rdf+xml rdf
text/turtle turtle
text/plain ntriples
application/ld+json json-ld

representation URL’s

The respective representations of the document describing an entity are directly retrievable by using following extensions:

Extension for a response format in
.html html
.rdf rdf
.ttl turtle
.nt ntriples
.jsonld json-ld

CORS

This server has CORS support. When one adds a Origin header to a HTTP GET request, the response will contain a Access-Control-Allow-Origin.

Reconciliation server

The Reconciliation API is a simple web service supporting the interlinking of other datasets to the KBO dataset. This API accepts a simple string e.g. label as input and returns a list of potential matching identifiers in ranked order.

This API implements almost completely the OpenRefine Reconciliation API of which you can find the information at: Ordnance Survey en OpenRefine.

With as result that this API can be used directly from within OpenRefine/ LODRefine.

Not implemented yet queries using: * a union of types * properties

This API can be found at : KBOdata reconcilitiation API.
